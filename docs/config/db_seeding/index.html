Database Seeding

When spawning environments, you might want your databases to be populated with data before starting to use the environment.
To allow this, Raftt supports database seeding that is defined as a part of the raftt.yml file.

Seeding Methodsâ€‹

Raftt currently supports three seeding methods -

  • A builtin method to load a .sql dump file into a PostgresSQL database
  • A builtin method to load a .archive dump file into MongoDB
  • A om user-created script that seeds the databases

PostgreSQLâ€‹

Load a database dump file as a PostgresSQL .sql file. To allow all team members to share the same initial database state, we recommend the dump file to be committed to the repo.
This file can be created by pg_dumpall or by running raftt data dump.

To configure seeding of a PostgresSQL database, configure it in the raftt.yml configuration file -

data:
  - service: db
    type: postgres
    user: postgres # optional. Default=root
    dump: dump.sql

MongoDBâ€‹

Load a database dump file as a MongoDB .archive file. To allow all team members to share the same initial database state, we recommend the dump file to be committed to the repo.
This file can be created by mongodump or by running raftt data dump.

To configure seeding of a MongoDB database, configure it in the raftt.yml configuration file -

data:
  - service: db
    type: mongodb
    dump: dump.archive

Custom Scriptâ€‹

You can create a custom script if you don't use PostgresSQL or MongoDB or want a more customized database seeding experience.
This allows complete control over how you wish to populate your databases, and is database type independent.
Using the custom script support, Raftt supports advanced use-cases such as:

  • Loading a large dump from S3
  • Auto-generating DB data
important

The DB process is not guaranteed to be ready to accept connections before the script is executed.
The script should wait for the DB and poll it.

To configure custom seeding of your database, configure it in the raftt.yml configuration file -

data:
  - service: db
    type: script
    script: db_seed.sh
    keyProvider: key.sh # optional

keyProviderâ€‹

Seeding the database using a seeding script can take a while, especially if it downloads large amounts of data. To save time, Raftt can fist check if the dump was already used.
Raftt checks the dump's unique key created by running the script supplied as keyProvider.
A simple example is a script that prints the hash of the dump file -

shasum dump.sql

Sample Codeâ€‹

The following code samples demonstrate how to poll different databases to make sure they are up and running, and then seed them.

  • postgresSQL

#!/usr/bin/env bash
set -e

DB_HOST=db
DB_USER=root
DB_PASSWORD=password
DB_NAME=app_development
DUMP_PATH=dump.sql

retry() {
    max_attempts="${1}"; shift
    seconds="${1}"; shift
    attempt_num=1

    until "${@}"; do
        if [ "$attempt_num" -eq "$max_attempts" ]
        then
            echo "Attempt $attempt_num failed and there are no more attempts left!"
            exit 1
        else
            echo "Attempt $attempt_num failed! Trying again in $seconds seconds..."
            attempt_num=$(( 1 + attempt_num ))
            sleep "$seconds"
        fi
    done
}

retry 1>&2 "5" "1" mysqladmin -h $DB_HOST -u $DB_USER -p"$DB_PASSWORD" ping

mysql -u $DB_USER -p"$DB_PASSWORD" -h $DB_HOST -e "DROP DATABASE IF EXISTS $DB_NAME;"
mysql -u $DB_USER -p"$DB_PASSWORD" -h $DB_HOST -e "CREATE DATABASE $DB_NAME;"
mysql -u $DB_USER -p"$DB_PASSWORD" -h $DB_HOST $DB_NAME < $DUMP_PATH

raftt data commandâ€‹

To explicitly initiate actions on the databases - dump, save, load, and seed, you can use the raftt data command.

Reloading a Serviceâ€‹

When reloading a service using the raftt reload, the default behavior is not to reseed the databases that are defined in raftt.yml, but keep the state as-is.
To reseed the databases when reloading a service them, use raftt reload -r
