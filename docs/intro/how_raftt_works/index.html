High Level Architectureâ€‹

Raftt's solution has two main modules - the client-side module and the backend module.

Client Side Moduleâ€‹

The client side module runs on your dev machine and includes two sub-modules - the CLI and the daemon.
The CLI is used by the users to run Raftt commands.
The daemon runs in the background and performs several tasks - syncing local files with the dev container, monitoring changes in the git status, communication with the backend etc.

The client self-updates automatically once a new version is available.

Backendâ€‹

Environments Managerâ€‹

This module manages the state machine of the different environments.
It spawned environments, takes them down, and puts them to sleep and wakes them up when necessary.

Key Conceptsâ€‹

Raftt's Containersâ€‹

Raftt runs your containers, but slightly modifies the container behavior to improve the development experience.
The actual main process of the container is Raftt's process called lifeguard, instead of the process defined by you in the Docker Compose file.
lifeguard runs your main process when it starts, but doesn't die when the process dies. This behavior allows much better debugging capabilities as it prevents the container from terminating and losing the state it had when crashing.
Keeping the container up also allows better development velocity, because you don't need to spawn a container every time you restart the main process.
Note that raftt stop and raftt restart only affect the user's main process, not the container itself, and raftt rebuild rebuilds and reloads the container itself.
